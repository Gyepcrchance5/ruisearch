# 飞书原生文档格式规范

调研报告走 feishu-mcp-pro 推送飞书文档时使用。核心原则：**少即是多——标准 Markdown 能表达的就不要用富文本块，只在信息需要被"看见"时才强化。**

## API 硬限制（必须遵守，不可绕过）

以下限制来自 feishu-mcp-pro 当前版本的实际行为，非审美偏好。写错会导致渲染异常：

| 限制 | 说明 |
| --- | --- |
| callout `background-color` | `doc_write` 时正常生效（`light-blue`/`light-yellow`），`doc_update` 时会被丢弃，修复需重新 `doc_write` 全文 |
| `{color}` 标题属性 | **不支持**。渲染为字面文本（如 `# 标题 {color="blue"}`） |
| `[bold]` 标签 | **不支持**。callout 内渲染为字面文本，用 `**bold**` 替代 |
| `doc_create` + `markdown` | 含复杂块时可能只建标题、正文丢失。**禁止**传 markdown 参数 |

## 何时使用

用户说"推送到飞书""生成飞书文档""飞书格式输出"时触发。

## 执行流程（必须两步，不可合并）

```
1. doc_create(title="报告标题")          → 拿到 document_id（不传 markdown）
2. doc_write(document_id, markdown=全文)  → 写入完整内容
```

两步走保证内容不丢。

### 文档密级

feishu-mcp-pro 不支持设置文档密级。生成后需在飞书网页端手动设置。

## 调试方法

写入后如果某个属性"看起来没生效"，先不要下结论。`doc_fetch` 默认 `format="markdown"` 不显示 block 级属性（如 `background-color`、`column-widths`），检查时改用：

```
doc_fetch(document_id, format="xml", detail="with-ids")
```

XML 里看对应 `<callout>` 或 `<table>` 节点是否有目标属性。确认后再判断是 API 不支持还是写法问题。

格式：`{核心结论一句概括}：{时间范围}{主题}调研`

要求：
- 前半句抓住最核心的发现或趋势，不写"关于""有关""分析"等废话
- 中文数字间不加空格
- 不用 `{color}`、不用 emoji
- 控制在 30 字以内

示例：
- ✅ 大模型军备竞赛：2026年7月AI行业调研
- ✅ 从产能地狱到现货促销：小米汽车2026年7月动态调研
- ❌ 2026 年 7 月 AI 行业动态：大模型密集发布季

## 核心规则

1. 正文、列表、标题正常写，不加花活
2. callout 只用于**核心结论**和**风险提示**，每篇最多 3 处
3. 彩色文字只用于**数据方向**（涨/跌/超预期/低于预期），不用于标题或标签
4. 表格用标准 MD 格式（`|` 分隔），仅在单元格需要彩色文字时用 `lark-table`
5. 不主动使用 grid 多栏布局，除非两件事确实需要并排对比
6. 不用 emoji 标题前缀，不用彩色标题

## 各内容块的约定

### 标题

纯文本，不加 `{color}`。

### 核心结论

放在报告顶部，一个 callout。**首句用加粗一句话概括最核心发现**（≤50字），需要展开的内容另起一段，总行数不超过6行。

```
<callout emoji="💡" background-color="light-blue">
**一句话核心发现。**

展开说明或补充背景，另一段。
</callout>
```

### 风险/不确定性提示

放在相关章节末尾：

```
<callout emoji="⚠️" background-color="light-yellow">
**一句话概括风险，加粗。**

如需进一步说明影响或建议，另起一段，控制在4行以内。
</callout>
```

### 高亮块使用规则

高亮块文字一旦过长，阅读体验急剧下降。**必须遵守：**

1. **首行必加粗**：第一句是结论句，用 `**粗体**` 突出，让人一扫就知道说什么
2. **内容分段**：需要讲两个以上要点时，用空行隔开，不用列表符号
3. **控制总量**：每个 callout ≤ 6 行（飞书渲染后的视觉行），核心结论不可超过核心规则能数的行数
4. **宁拆不塞**：内容超过 6 行 → 说明不该放 callout，拆成"小标题 + 正文段落"
5. **不要嵌套**：callout 内不要再包 callout、表格、列表。只有加粗文字和纯文本段落

### 表格

标准 MD 表格即可（`|` 分隔），飞书正常渲染。只在**单元格内需要彩色文字**（如涨跌数据）时才用 `lark-table`，因为标准 MD 表格不支持 `<text color>`。

**列宽规则（仅 lark-table）：**

- 所有列宽之和控制在 800-900px
- 文字列 ≥ 200px，日期/数字等短列 70-90px
- 4 列表示例：`column-widths="200,80,260,300"`
- 3 列表示例：`column-widths="220,280,350"`
- 2 列表示例：`column-widths="200,650"`

### 加粗

用于以下三处，正文其余部分不主动加粗：1) 表格表头；2) callout 首句结论；3) 正文段落中的**关键结论句**（每段最多一处）。

### 彩色行内文字

仅用于标注**数据方向**，不用于给标题或标签"上色"：

- `<text color="green">+15%</text>` — 增长、超出预期
- `<text color="red">-8%</text>` — 下滑、低于预期

禁止：给章节标题标色、给"利好""利空"等定性词标色、给平台名称标色。

### 引用块

来源标注放在报告末尾，用 `>`：

> 数据来源：Exa → 钜亨网、36氪；腾讯行情 API

### 列表

标准 Markdown 列表即可，不嵌套超过两级。

## 不需要用的能力

以下功能**不主动使用**，除非用户明确要求：

- grid 多栏布局 — 标准纵向排版已经足够
- 彩色 callout（红/紫/橙等）— 只有蓝（结论）和黄（风险）两种
- `<text>` 颜色标签为标题着色
- emoji 前缀装饰（标题前不加 ⭐📌 等图标）
- `<quote-container>` — 用标准 `>` 引用即可
- `<whiteboard>` 流程图 — 需要时用户会明确说

## 生成后检查

- [ ] doc_create 未传 markdown，只设 title
- [ ] 标题格式：核心发现 + 时间范围 + 主题，≤ 30 字，无空格无 emoji
- [ ] callout 每篇 ≤ 3 处，首句均加粗，每个 ≤ 6 行，无嵌套
- [ ] 彩色文字仅用于涨跌/数据方向
- [ ] 标题纯文本，无 `{color}`
- [ ] 表格：无彩色文字时用标准 MD 表，有 `<text color>` 时用 lark-table，列宽之和 800-900px
- [ ] 无多余装饰性元素
