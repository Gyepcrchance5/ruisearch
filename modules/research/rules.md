# 调研规则
## 调研模式
### 快速查询（默认）
用户说"查一下""帮我搜""XX是什么"时，走快速路径：
1. 单次搜索（Exa 首选，失败则 WebSearch）
2. chat 内直接回答，标注来源和置信度
3. 不生成文件
### 完整报告
用户说"调研""分析报告""整理报告"时，走完整流程：
**搜索两遍**
- 第一遍：广泛搜索，覆盖主要关键词，获取初步信息
- 第二遍：换角度补充，验证关键数据点
**分析一遍**
- 分类整理（按品类/时间/区域等维度）
- 检查逻辑自洽，标注置信度
**输出一遍**
- 按选定格式整理（默认飞书紧凑格式，详见 `../document/feishu_compact.md`）
- 标注来源和时间
## 搜索工具链（优先尝试，平滑降级）
### 工具列表
**1. Exa 语义搜索（首选）**
```bash
# 网页搜索 — 技术和行业调研首选
mcporter call 'exa.web_search_exa(query: "关键词", numResults: 5)'
# 代码搜索 — 技术实现细节
mcporter call 'exa.get_code_context_exa(query: "代码问题", tokensNum: 3000)'
```
Exa 是高质量 AI 搜索引擎，擅长英文内容和技术文档。核心特点：
- 语义理解而非关键词匹配，对技术术语和行业概念识别准确
- 结果质量显著高于通用搜索引擎
- 支持代码上下文搜索（`get_code_context_exa`）
- 默认每次返回 5 条，可调高但建议 ≤10（平衡质量和速度）
使用场景对照：
| 场景 | 命令 | numResults |
| --- | --- | --- |
| 技术调研/行业分析 | `web_search_exa` | 5 |
| 竞品调研/产品对比 | `web_search_exa` | 5-8 |
| 代码/实现细节 | `get_code_context_exa` | tokensNum: 3000 |
| 快速事实查证 | `web_search_exa` | 3 |
**2. WebSearch（自动降级）**
内置搜索工具。Exa 调用失败（超时 10s 或返回错误）时直接使用，无需重试 Exa，无需标注警告。
**3. 专利检索（技术调研/竞品分析时自动启用）**
专利数据是技术调研和竞品分析的重要数据源。根据查询类型选择不同方式：
- 专利号精准查询：WebFetch(`https://patents.google.com/patent/{专利号}/zh`)，提取标题、摘要、权利要求、法律状态、同族
- 关键词查专利：WebSearch("{关键词} 专利 site:patents.google.com")
- 竞品专利分析：WebSearch("{公司名} 专利布局 {技术领域}")
详见下方「专利检索规则」章节。
**4. Jina Reader（URL 全文读取）**
```bash
curl -s "https://r.jina.ai/URL"
```
读取搜索结果中的指定网页全文。用于深度分析关键页面内容。
### 降级规则
- mcporter 命令不存在：**先尝试** `npm install -g mcporter`，安装失败再降级 WebSearch
- mcporter 存在但 Exa 未注册：**先尝试** `mcporter config add exa --url https://mcp.exa.ai/mcp && mcporter auth exa`，配置失败再降级 WebSearch
- Exa 认证过期：**先尝试** `mcporter auth exa`，仍失败再降级 WebSearch
- Exa 已配置但单次调用失败（超时/网络错误）：直接降级 WebSearch，不去重试
- Jina Reader 用于读取已知 URL，不作为独立搜索工具
### 专利检索规则
**何时自动启用**
当调研类型为以下任一且涉及技术细节时，自动启用专利检索：
- 技术调研（涉及技术路线、技术方案、技术壁垒）
- 竞品分析（涉及竞品技术布局、核心能力）
- 供应商调研（涉及供应商技术实力、专利壁垒）
- 关键词触发：专利、查新、技术壁垒、核心方案、知识产权、IP
**检索策略**
| 查询类型 | 识别方式 | 检索方式 | 返回信息 |
| --- | --- | --- | --- |
| 专利号查询 | 匹配`^[A-Z]{2}\d+` | WebFetch Google Patents | 标题+摘要+权利要求+法律状态+同族 |
| 技术关键词 | 其他 | WebSearch+CNIPA爬虫 | 专利列表+摘要 |
| 竞品专利 | 公司名+专利 | WebSearch | 专利布局概况 |
**Google Patents 页面解析要点**
WebFetch Google Patents 页面后，从返回的 markdown 中提取：
- 标题：通常在页面最顶部
- 摘要：在 "Abstract" 或 "摘要" 区域
- 权利要求：在 "Claims" 或 "权利要求书" 区域
- 法律状态：在 "Legal Events" 或 "法律事件" 区域，关注授权/失效/在审状态
- 同族：在 "Family" 或 "同族" 区域，列出同族专利号
- 申请人/发明人：在页面基本信息区域
**置信度处理**
- Google Patents 数据：高置信度（数据源权威）
- CNIPA爬虫数据：中置信度（仅返回摘要，信息有限）
- WebSearch 专利信息：中低置信度（需交叉验证）
- 多个来源不一致时：以 Google Patents 为准
## 股票行情查询
### 适用场景
用户说"查一下股价""港股收盘""A股行情""XX股票涨跌"时，使用腾讯行情 API 直接获取实时数据。
### 接口说明
腾讯免费行情接口，无鉴权，延迟 3-5 秒。
```bash
# 单只
curl -s "https://qt.gtimg.cn/q=sh600519" | iconv -f GBK -t UTF-8
# 批量
curl -s "https://qt.gtimg.cn/q=sh600519,sz000858,r_hk01810" | iconv -f GBK -t UTF-8
```
### 代码格式
- A 股上海：`sh` + 6 位代码（如 sh600519 茅台）
- A 股深圳：`sz` + 6 位代码（如 sz300750 宁德时代）
- 港股：`r_hk` + 5 位代码（如 r_hk01810 小米）
### 返回字段（按 `~` 分隔，关键字段索引）
- [1] 名称、[2] 代码、[3] 当前价、[4] 昨收、[5] 今开
- [6] 成交量、[30] 时间戳、[31] 涨跌额、[32] 涨跌幅
- [33] 最高、[34] 最低、[37] 成交额
- 港股行末尾有 HKD 货币标识；A 股为 CNY
### 常用指数代码
- 恒生指数：`r_hkHSI`
- 恒生科技：`r_hkHSTECH`
- 上证指数：`sh000001`
- 深证成指：`sz399001`
- 创业板指：`sz399006`
### 输出规范
直接解析 API 返回数据，在 chat 内以表格形式展示，包含：股票名称、代码、当前价、涨跌额、涨跌幅。批量查询时按涨跌幅排序。
### 消息面调研
如果用户追问"为什么涨/跌"，切换到 ruisearch 的 Exa 搜索模式，搜索"{股票名} {日期} 利好/利空 消息面"，在 chat 内直接回答并标注来源和置信度。
## 关键词组合模板
### 产品调研
```
品牌 + 产品类型 + 特征
```
示例：
- "大疆 无人机 避障技术"
- "特斯拉 Model Y 悬架结构"
- "索尼 A7CR 传感器规格"
### 技术调研
```
技术名称 + 行业/应用 + 时间
```
示例：
- "固态电池 量产进展 2025"
- "激光雷达 车规级 成本"
- "大语言模型 Agent框架"
### 供应商调研
```
供应商名 + 产品类别 + 技术特点
```
示例：
- "宁德时代 储能电池 能量密度"
- "博世 线控制动 冗余方案"
- "TI 电源管理芯片 车规级"
### 市场分析
```
市场细分 + 区域 + 年份
```
示例：
- "中国 新能源汽车 2025 市场份额"
- "全球 AI芯片 2025 竞争格局"
- "东南亚 电动车 充电桩 覆盖率"
### 专利调研
```
技术领域 + 申请人/竞争对手 + 时间
```
示例：
- "DMS 驾驶员监测 麦格纳 专利"

## 调研经验（通用启发式规则）
### 概念等价需逐项验证
- 不同术语之间不一定是等价关系（如"快充"可能指不同协议和功率等级）
- 需要逐产品/逐场景确认，不能仅凭名称推断等价
- 用户提供纠正信息时，立即修正文档
### 官方来源优先级
- 品牌官网/官方文档 > 行业报告/权威媒体 > 论坛/社交媒体
- 官方配置表、技术白皮书、发布会内容为最高优先级来源
- 官方未公开的信息不能直接确认，需降低置信度
### 区分相似概念
- 相似术语可能指向不同技术/产品（如"混合动力"有多种架构）
- 搜索时注意区分，避免混淆不同概念
- 对不确定的区分，逐项单独搜索验证
### 批量搜索策略
- 先搜官方/一手来源确认，再搜媒体报道补充
- 同一类别/品牌可合并搜索提高效率
- 对未确认的项目，降低置信度标注，而非直接确认
## 输出规范
### 报告架构（完整报告模式）
报告由**必选骨架**和**可选模块**组成，根据调研主题和用户意图灵活组合。
**必选骨架（3+1章）**
| 章节 | 内容 | 说明 |
| --- | --- | --- |
| 市场概况 | 规模、增速、应用结构 | 回答"多大" |
| 竞争格局 | 份额、主要玩家、技术路线 | 回答"谁在做" |
| 技术趋势 | 当前主流技术+下一代方向 | 回答"往哪走" |
| 关键结论 | 核心发现汇总 | 必须用`>`引用块收尾 |
**可选模块（按需插入）**
| 模块 | 触发条件 |
| --- | --- |
| 产业链/成本结构 | 用户问成本/供应链/国产化/供应商时加入 |
| 应用场景深挖 | 用户有明确关联领域时加入（如"跟后视镜的关系"） |
| 专利布局 | 技术调研、竞品分析时**默认加入**（搜索专利数据源）；纯市场调研时按需加入 |
| 财务/估值分析 | 用户关注投资方向/财务对比时加入 |
**规则**：必选骨架保证报告下限，可选模块和自定义章节按调研内容灵活增减。不要为了填充而加入空洞章节。
### 表格格式（遵循飞书表格密度控制）
所有表格**必须**遵循 `../document/feishu_compact.md` 的「表格密度控制」规则：
- 不超过 4 列
- 每格不超过 30 字
- 长文本描述用两层结构：表格 + 展开
### 详细分析表格
需要展开详细内容的分析结果，按两层结构输出：
1. 表格层：名称/编号/日期/状态等短字段（4列）
2. 展开层：每条记录用 **加粗标题** + **要点列表** 展开详细内容
禁止将长文本塞入表格单元格。
### 置信度标注
| 级别 | 标准 | 标注方式 |
| --- | --- | --- |
| 高 | 官方数据/多来源一致 | 直接使用 |
| 中 | 单一来源/逻辑推断 | 标注"待验证" |
| 低 | 网络传言/信息矛盾 | 标注"需进一步确认" |
### 来源标注
每条信息在表格中增加"信息来源"列：
| 项目 | 特点 | 信息来源 |
| --- | --- | --- |
| 产品A | 核心参数 | 官网技术文档 |
| 产品B | 市场表现 | 行业报告 |
